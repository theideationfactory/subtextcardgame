# Subtext App – App Store Readiness Evaluation

**Date:** 2025-12-30  
**App:** Subtext (Expo / React Native)  
**Bundle ID (iOS):** `com.adamkretz.subtext`  
**Config Source:** `app.json`, `/app`, `/contexts`, `/supabase/functions`, `/services`

---

## 1. High-Level Summary

Overall, the Subtext app is **technically close to App Store–ready**, but there are several **policy/compliance gaps** that are very likely to block approval if not addressed.

**Hard blockers (must fix before submission):**

- **No in-app or hosted Privacy Policy**
- **No Terms of Service / EULA surfaced to users**
- **No working in-app account deletion flow**, despite account creation
- **No real reporting/blocking tools for user-generated content (chat)**

**Likely review friction / questions:**

- Declared **microphone** and **background processing** capabilities with no visible in-app usage
- NFT / crypto integrations need to be clearly **optional** and **not used as payment rails**
- AI/OpenAI usage needs to be properly disclosed in **privacy** and **App Store privacy questionnaire**

The sections below map key areas to Apple’s guidelines and recommend concrete code/config changes.

### 2.2 Permissions Declared vs Used

  - `NSMicrophoneUsageDescription`

  - `CAMERA`, `READ_EXTERNAL_STORAGE`, `WRITE_EXTERNAL_STORAGE`, `RECORD_AUDIO`

**Actual usage in code (grep + file review):**

- **Camera / Photos:**
  - `expo-image-picker` is actively used in card/deck creation screens:
    - `app/(tabs)/card-creation-new.tsx`
    - `app/(tabs)/deck-creation.tsx`
    - `app/(tabs)/deck-detail.tsx`
  - Requests media library permission, uses selected images for cards/decks.
- **Microphone / Audio:**
  - No use of `expo-av`, `Audio.Recording`, or similar audio APIs was found.
  - No visible UI for audio recordings.
- **Background fetch / processing:**
  - `expo-background-fetch` and `expo-task-manager` are declared as plugins.
  - No code using `BackgroundFetch` or `TaskManager` was found.
- **Face ID:**
  - `expo-secure-store` plugin is configured with a Face ID permission string.
  - `lib/supabase.ts` uses a custom SecureStore adapter for auth storage; this may indirectly use biometrics.

**Risks & Recommendations:**

- **Microphone:**
  - *Risk:* Apple often flags unused sensitive permissions.
  - **Recommendation:** If you are not actually recording audio, **remove** `NSMicrophoneUsageDescription` and the Android `RECORD_AUDIO` permission.

- **Background fetch/processing:**
  - *Risk:* Declaring `UIBackgroundModes` without an actual feature can trigger questions.
  - **Recommendation:** If you are not running background tasks, **remove** `UIBackgroundModes` entries and the `expo-background-fetch` / `expo-task-manager` plugins from `app.json`.

- **Face ID:**
  - *Risk:* Low. Secure token storage w/ Face ID is acceptable.
  - **Recommendation:** Ensure any UI that relies on biometrics clearly describes this behavior in your privacy policy.

## 3. Authentication, Accounts, and Deletion

**Files reviewed:**

- `contexts/AuthContext.tsx`
- `app/login.tsx`
- `app/_layout.tsx`
- `app/welcome.tsx`
- `app/(tabs)/settings.tsx`

### 3.1 Sign-in & Guest Mode

- **Login screen:** `app/login.tsx`
  - Email/password sign-in and sign-up using Supabase.
  - `Continue as Guest` uses `signInAnonymously()` in `AuthContext`.
- **Routing:** `app/_layout.tsx`
  - Public routes: `welcome`, `login`, `drafts`, `inbox`, `chat`, `upgrade-account`.
  - Authenticated routes: `(tabs)` stack.
- **Sign in with Apple / 3rd party:**
  - No Google/Facebook sign-in.
  - Apple Sign In entitlement is declared, but no `AppleAuthentication` usage in the code.

**Assessment:**

- Apple **requires Sign in with Apple only when using other third‑party sign-in providers**. Email/password + guest mode is typically acceptable.
- Current setup is **compliant** on that point, though the unused Apple Sign In entitlement could be removed for cleanliness.

### 3.2 Account Deletion (Guideline 5.1.1(v))

- Settings screen: `app/(tabs)/settings.tsx`
  - Has a **"Delete Account"** button, but **no handler** attached (no deletion logic).
- No Supabase function or edge function was found that deletes a user account.

**Assessment:**

- Apple’s current rules **require a way to initiate account deletion from within the app** for apps that create accounts.
- Current implementation is **non-compliant**.

**Required Changes:**

1. **Implement backend deletion support** (Supabase):
   - Create a **Supabase Edge Function** (e.g. `delete-account`) that:
     - Authenticates the current user from the JWT.
     - Uses the service role key to call `auth.admin.deleteUser(user.id)`.
     - Cleans up or anonymizes related data (cards, messages, spreads) per your data retention policy.

2. **Wire up the Delete Account button** in `settings.tsx`:
   - Show a confirmation dialog.
   - Call the `delete-account` function.
   - On success, sign the user out and navigate to `welcome` or `login`.
   - Display clear messaging about the **effect and timing** of deletion (immediate vs. queued).

3. **Document deletion** in the Privacy Policy / Terms:
   - Explain what data is deleted and what may be retained (e.g., anonymized analytics, logs).

## 4. User-Generated Content, Chat, and Safety

**Files reviewed:**

- `app/chat.tsx`
- `services/chatService.ts`
- Friend/conversation RPC usage

### 4.1 Chat & UGC Features

- Direct user-to-user chat with:
  - Text messages
  - Shared cards and spreads (via `message_type` and `card_id`/`spread_id`).
- Real-time updates via Supabase `postgres_changes` channel subscriptions.
- Speech-act analysis and response strategies using OpenAI via Supabase functions.

### 4.2 Moderation / Reporting / Blocking

- `chat.tsx` provides a **long-press action toolbar** with buttons labeled:
  - `Request`
  - `Reveal` (flag icon)
  - `Other`
- `Reveal` currently shows a **placeholder** alert ("Reveal message functionality coming soon!").
- `ChatService` has:
  - `deleteMessage()` – allows sender to soft-delete **their own** messages only.
  - No functions for **reporting other users/messages**.
  - No **block/mute** functionality.

**Guideline impact:**

- Apple requires that apps with user-generated content provide:
  - A way to **report offensive content**.
  - A means to **block abusive users**.
  - Moderation mechanisms to **remove objectionable content**.

**Current state:**

- There is **no real reporting pipeline** and **no block/mute** system.
- This is a **likely blocker** for review, especially because chat is a central feature.

**Required / Recommended Changes:**

1. **Implement message reporting:**
   - Add a `message_reports` table in Supabase (reporter_id, message_id, reason, created_at, status).
   - Wire the `Reveal` action in `chat.tsx` to open a simple **Report** dialog:
     - Predefined reasons (Harassment, Hate, Spam, Other).
     - Optional free-text notes.
   - Insert reports into `message_reports` and build an admin view (web or dashboard) for moderation.

2. **Implement user blocking:**
   - Add a `blocked_users` table (user_id, blocked_user_id, created_at).
   - On send/fetch of messages and friend invites, enforce blocking rules (no new messages if blocked).
   - Provide UI in chat or settings to **Block user** and **Unblock user**.

3. **Content guidelines / code of conduct:**
   - Add a simple **"Community Guidelines"** section in-app (e.g., in Settings or onboarding).
   - Reference these in Terms of Service.

## 5. AI & OpenAI Usage

**Files reviewed:**

- `supabase/functions/analyze-speech-acts/index.ts`
- `services/chatService.ts`
- `app/chat.tsx`
- `app/(tabs)/card-creation-new.tsx`
- Deck/Detail AI image generation calls (Supabase functions)

### 5.1 Speech Act Analysis & Response Strategies

- Edge function `analyze-speech-acts`:
  - Uses `OpenAI` SDK and `gpt-4o-mini`.
  - Parses model output and writes `speech_acts` back to `messages` table.
- `ChatService.analyzeSpeechActs()` and `generateResponseStrategies()` call Supabase functions.
- Frontend provides a **toolbox** with tags summarizing speech acts and AI-generated response strategies.

### 5.2 AI Image Generation

- `card-creation-new.tsx` uses Supabase functions for image generation.
- `deck-creation.tsx` and `deck-detail.tsx` call a `generate-card-image` function for deck/option images.

**Guideline impact:**

- Generative AI is allowed but must **not produce prohibited content** (hate, explicit, etc.).
- Must be clearly disclosed in privacy and data-usage documentation.

**Recommendations:**

1. **Privacy Policy:**
   - Explicitly state that text and content may be sent to **OpenAI** for:
     - Speech-act analysis of messages.
     - Image generation.
   - Explain what data is sent (e.g., message text, card names/descriptions) and retention policies.

2. **Content safety:**
   - Consider adding server-side filtering of prompts and/or OpenAI safety settings.
   - Add UI copy indicating that generated images/content may be imperfect and should follow community guidelines.

## 6. NFTs, Wallets, and Crypto

**Files reviewed:**

- `app/utils/nftMinter.ts`
- `app/services/walletService.ts`
- `app/components/NftMintingButton.tsx`
- `app/(tabs)/settings.tsx`

### 6.1 Current Implementation

- `nftMinter.ts`:
  - Integrates with **Alchemy**, **Pinata**, and an NFT contract (`SubtextNFTABI`).
  - Handles IPFS uploads, metadata, and NFT mint transactions.
- `walletService.ts`:
  - Placeholder for WalletConnect.
  - A **private-key based** connection helper clearly labeled as **testing only**.
- `NftMintingButton.tsx`:
  - Uses `SubtextNftMinter` but currently only simulates minting with alerts and logs.
- `settings.tsx`:
  - "Connect Wallet" button is currently a **simulated connection** (mock address generation).
  - "Mint Card as NFT" button navigates users to cards with an informational alert.

**Assessment:**

- NFTs are **optional** and currently not tied to in-app purchases.
- There is **no on-device private key storage** in production code (the testing helper is present but not hooked to UI).

**Guideline impact:**

- Apple allows NFTs and blockchain **as long as they are not used to avoid IAP** for digital goods.
- You must ensure that:
  - Users are not buying NFTs inside the app as a way to unlock features.
  - Any external payments occur through approved crypto wallets outside Apple’s payment system.

**Recommendations:**

1. **Keep NFT minting optional** and not required for core gameplay.
2. **Document NFTs clearly** in Terms of Service:
   - No guarantee of value.
   - Gas fees and risks are borne by the user.
3. Double-check that **no private keys or secrets** are stored or passed in the mobile app.

## 9. Legal Docs: Privacy Policy & Terms of Service

**Findings:**

- `grep` indicates **no in-app screen or link** for:
  - Privacy Policy
  - Terms of Service / Terms of Use / EULA
- An older internal document (`SWE12.30.25AppStoreEvaluation.md`) flags this as missing as well.

**Guideline impact:**

- Apple **requires** a **Privacy Policy URL** for apps that:
  - Use account registration.
  - Access certain device features (camera/photos).
  - Connect to external services and backends.
- Terms of Service are **strongly expected** for apps with accounts, UGC, and NFTs.

**Required Actions:**

1. **Draft external documents:**
   - **Privacy Policy** covering:
     - Data collected (account info, cards, spreads, chat messages, images).
     - How data is used (core functionality, AI processing via OpenAI, storage with Supabase, NFTs via Pinata/Alchemy).
     - Third parties involved and links to their policies.
     - Data retention and deletion.
   - **Terms of Service / EULA** covering:
     - User responsibilities, prohibited conduct.
     - Ownership of cards, spreads, and generated content.
     - NFT-related disclosures and lack of guarantee of value.
     - Limitation of liability, governing law, and dispute resolution.

2. **Host these documents** on a stable URL (e.g., your marketing site or a simple static site).

3. **Surface them in-app:**
   - Add a **"Legal"** or **"About"** section in `app/(tabs)/settings.tsx` with buttons:
     - "Privacy Policy" (opens WebBrowser or in-app screen).
     - "Terms of Service".

4. **Populate App Store Connect metadata:**
   - Privacy Policy URL.
   - Support URL.
   - Marketing website URL (if available).

## 12. Overall Assessment

The Subtext app has a strong technical foundation and a clear value proposition (card creation, spreads, friend-based chat, optional AI and NFTs). From an engineering perspective, it is very close to being ready for the App Store.

However, **legal and policy compliance work** remains outstanding:

- Without a **Privacy Policy**, **Terms of Service**, **account deletion**, and **UGC safety tools**, the app is at high risk of rejection.
- Once these pieces are implemented and wired into the code paths identified above, the app should be in a solid position for submission.
